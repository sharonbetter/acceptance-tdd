# 验收测试驱动 + TDD 全周期

**先规划任务；先验收、后实现；验收文档成稿后先交用户确认，再动结构与产码；未全绿不宣称完成。**

---

## 一、ATDD 与 TDD 双循环模型

### 核心理念

| 层次 | 名称 | 面向 | 聚焦 | 红绿节奏 |
|------|------|------|------|----------|
| **外循环** | ATDD（验收驱动） | 业务与需求 | 端到端验收场景 | 🔴 红：验收测试失败 → TDD 实现 → 🟢 绿：验收测试通过 |
| **内循环** | TDD（测试驱动） | 设计与实现 | 代码逻辑、类和方法 | 🔴 红：单元测试失败 → 写极简实现 → 🟢 绿：单元测试通过 |

**形象比喻**：ATDD 像导航设定目的地（从 A 到 B），TDD 像方向盘微调（每秒钟根据路况调整 2 度）。没有 ATDD，TDD 可能走向错误终点；没有 TDD，ATDD 会因缺乏可维护的底层实现而崩塌。

### 双循环流程图

```
┌─────────────────────────────────────────────────────────┐
│                    ATDD 外循环                           │
│  ┌──────────────┐    ┌──────────────┐                  │
│  │  编写验收场景  │ → │  运行验收测试  │ → 🔴 失败（红）   │
│  │  (Given-When-Then) │               │                  │
│  └──────────────┘    └──────────────┘                  │
│         ↓                                            │
│  拆解为多个 TDD 任务 ─────────────────────────────────→ │
│         ↓                                            │
│  ┌──────────────────────────────────────────────────┐  │
│  │              TDD 内循环（每个任务）                 │  │
│  │  🔴红 → 验红 → 🟢绿 → 验绿 → 重构                   │  │
│  │  （单元测试级别）                                   │  │
│  └──────────────────────────────────────────────────┘  │
│         ↓  重复直到所有任务完成                        │
│  运行所有单元测试 → 全绿                               │
│         ↓                                            │
│  再次运行验收测试 → 🟢 通过（绿）                      │
│         ↓                                            │
│  重构并重复 → 进入下一个验收场景                        │
└─────────────────────────────────────────────────────────┘
```

### ATDD → TDD 拆解原则

1. 拿到一个验收场景后，思考「为了让这个验收测试通过，需要哪些组件/类/方法」
2. 将大任务拆解成若干个**最小颗粒度**的 TDD 任务
3. 每个 TDD 任务独立执行🔴红→🟢绿→重构循环
4. 所有内循环任务完成后，运行所有单元测试 → 全绿 → 再跑外循环验收测试

### ATDD 驱动协作接口，TDD 驱动实现细节

ATDD 从验收测试的步骤中提炼出系统外部的协作接口（高层次接口骨架），TDD 再驱动模块内部的实现细节（算法、条件分支、异常处理）。ATDD 和 TDD 都是先红再绿的过程。

**示例：用户注册功能的 ATDD → TDD 驱动过程**

#### 步骤 1：编写验收测试（Gherkin 风格）

```gherkin
Feature: 用户注册
  Scenario: 使用有效邮箱和密码注册
    Given 未注册的用户邮箱 "test@example.com"
    When 用户提交注册请求，密码为 "Abc12345"
    Then 系统返回注册成功消息
    And 系统向该邮箱发送验证邮件
```

#### 步骤 2：将描述转化为自动化验收测试代码（骨架）

```java
// 注意：此时 UserRegistrationSteps 类完全是空的，只有方法签名
public class UserRegistrationSteps {
    @Given("未注册的用户邮箱 {string}")
    public void 未注册的用户邮箱(String email) {
        // 空实现 —— 这个空骨架就是驱动点
    }

    @When("用户提交注册请求，密码为 {string}")
    public void 用户提交注册请求密码为(String password) {
        // 空实现
    }

    @Then("系统返回注册成功消息")
    public void 系统返回注册成功消息() {
        // 空实现
    }

    @And("系统向该邮箱发送验证邮件")
    public void 系统向该邮箱发送验证邮件() {
        // 空实现
    }
}
```

#### 步骤 3：ATDD 驱动代码生成的逻辑

为了填写这些空方法，需要真实的类和方法，这自然驱动出以下设计决策：

| 验收步骤 | 驱动出的设计 |
|----------|-------------|
| Given "未注册的用户邮箱" | `UserRepository` 类，含 `existsByEmail(email)` 方法 |
| When "用户提交注册请求" | `RegistrationService` 类，含 `register(email, password)` 方法 |
| Then "系统返回注册成功消息" | `RegistrationResult` 枚举或对象，作为返回值类型 |
| And "系统向该邮箱发送验证邮件" | `EmailService` 类，含 `sendVerificationEmail(email)` 方法 |

#### 步骤 4：进入 TDD 内循环，驱动细节代码

对 `RegistrationService.register()` 方法进行 TDD（🔴红→🟢绿→重构）：

1. **🔴 红**：写单元测试 `shouldReturnSuccessWhenNewUserRegisters()`，确认失败
2. **🟢 绿**：写极简实现——调用 `userRepository.existsByEmail()` 检查，若不存在则创建用户，最后调用 `emailService.sendVerificationEmail()`
3. **重构**：优化代码结构，测试保持通过

**关键点**：
- ATDD 驱动了**模块间的协作协议**（高层次的接口骨架）
- TDD 驱动了**模块内部的实现细节**（算法、条件分支、异常处理）
- ATDD 和 TDD 都是先红再绿的过程，只是红绿发生在不同层次

## 二、工作流（按顺序，可迭代）

### 迭代规则

- **迭代触发**：任意步骤发现前置条件不满足、或验收文档需修订时，迭代回对应步骤
- **迭代方向**：见「循环回跳规则」

### 循环回跳规则（汇总）

| 失败发生的位置 | 回到哪一步 |
|----------------|-----------|
| ATDD 验收场景失败（步骤 6） | 回到步骤 5，识别遗漏的 TDD 任务，补完后再次验证 |
| TDD 单元测试失败（步骤 5） | 回到步骤 5 对应的 TDD 任务红阶段，逐任务修复 |
| 验收文档闸门未通过（步骤 3.1） | 回到步骤 3，重新澄清 |
| 需求变更导致任务规划失效 | 回到步骤 1，重新规划 |

### 通用纪律

- **数据不确定性先澄清**：在任何步骤遇到测试数据（账号、订单号、第三方返回值等）不确定时，先向需求方或用户澄清，再写入用例或配置，不得自行假设。

### 1. 任务规划（硬性前置）

拿到需求后，**先不深入细节**，只规划本轮任务范围与拆分：

1. **识别范围**：从 spec 中划出本轮要做的功能边界
2. **列出任务**：按顺序写出本轮所有任务（见下方示例）
3. **逐项执行并验证**：按任务清单顺序执行，每完成一项在清单中打勾；发现偏差则修正任务清单后再继续

**任务清单格式示例**：

```
## 本次任务清单
- [T1] 任务规划（划范围、列任务）
- [T2] 深度理解需求（识别 AI 假设 + 待确认问题）
- [T3] 逐项收窄待确认问题（2.1 逐项推进）
- [T4] 编写 ATDD 验收场景主表
- [T5] 验收文档交用户确认（闸门）
- [T6] 结构设计
- [T7] ATDD 拆解 → TDD 内循环执行
- [T8] ATDD 验收场景全绿验证
- [T9] 收工前自审
```

### 2. 深度理解 + 需求澄清

**先规划、后理解**——拿到需求后，先完成步骤 1 任务规划，再进入本步骤深度理解。

通读手头 **spec、需求、技术说明**，进行深度理解，区分两类情况：

| 类型 | 处理方式 |
|------|----------|
| AI 可自行合理推断 | 用 `[假设]` 标注，写入验收文档，不阻塞 |
| 影响实现方向、有多条可能路径的歧义/缺失 | 列入「待确认问题」（2.1），阻塞推进 |

**判断标准**：若不澄清会导致实现方向错误，且 AI 无法自行推断，则列为阻塞项。

**执行记录留痕（硬性）**：凡**新建**或**实质修订**主验收文档（表 A/B、主表、映射、Then 等），代理**不得**在主文内维护执行引用大表。须在 **配套执行记录文件**（`*.execution-log.md`，与主稿同目录、同主文件名）中追加，约定见 **[references/atdd-checklist-template.md](references/atdd-checklist-template.md)**「执行记录（配套文件）」节：文件名为 **`<主文件名>.execution-log.md`**（例：`feature-acceptance.md` → `feature-acceptance.execution-log.md`）；主稿正文**仅**保留一行指向该文件的链接。该文件内的表**追加**本会话**真实打开并对照过**的路径：**须含**实际用过的技能包内 `SKILL.md` 与 `references/**`（未打开则不写）；**若**为对齐 Then 还打开了 **项目 spec、主验收稿、集成/ E2E 源文件**，一并写入，用途列注明「非技能包」。**禁止**冒充未读文件；**禁止**要求用户代填。

### 2.1 待确认问题（歧义 / 缺失 / 易争议）

凡需求存在 **歧义、缺失或易引发争议** 的点，且 AI 无法自行合理推断的，**必须单独成表**，不得混在已写死的 Then 里假装已澄清。

- 使用 **[references/atdd-checklist-template.md](references/atdd-checklist-template.md)** 中的 **「待确认问题」** 与 **「待确认 → 测试映射」** 两表；**一行一问题**，`Q-ID` 全程稳定可追溯。  
- **用户视角发问（硬性）**：表内「澄清问句」**不得**只写实现视角（如「`PUT` 是否返回 400」）。须写清 **终端用户或业务读者能代入的情境**（「我正在…」「若我…」），再落到可选方案或判据；技术名词可放在括号里作对照。问句须**可直接复制**给产研/业务复述。  
- **逐项确认 + 自动带出下一项（硬性）**：与用户对话时，**每次只主动推进一项**待确认（默认顺序：**表 A 阻塞项**按 `A-xx` 升序 → **表 B** 按 `B-xx` 升序；若存在依赖，以依赖先决项优先）。用户给出可执行的结论后，代理须**同一轮内**：(1) 把结论写入验收文档（表 A/B、映射表、相关场景 **Then**、必要时「已决议」归档）；(2) **在回复末尾**自动附上**下一项**的、同样**用户视角**的可复制问句（若队列已空，则明确说明「待确认项已清」，并引导进入 **3.1 验收文档交用户确认（闸门）**）。  
- **便于审计的答复格式**：鼓励用户答复时带 **`Q-ID`**（例如 `B-01：改为方案 2`），代理回填时原样记入「结论摘要」。  
- **答复与结论**填回后，须同步改对应验收场景的 **Then**，并补全 **接口/E2E 映射** 列，便于直接落成自动化用例名与断言。

### 3. ATDD：验收测试点（Given / When / Then）

每条验收场景须用 **Given / When / Then** 编写，且**三者缺一不可**。主表（见模板）中每行还须具备：

| 项 | 须写清的内容 |
|----|----------------|
| **场景名称** | **硬性**：一两句**面向产品/业务读者**的可读短名（说明「验什么」），与 **场景 ID** 并列；**禁止**用 `S-01` 代替名称、也禁止名称与 **Then** 完全同文重复无区分度。 |
| **场景 ID** | 稳定编号（如 `S-01`），供映射表、用例名、对话引用。 |

| 段 | 须写清的内容 |
|----|----------------|
| **Given** | **前置条件**：系统/用户/数据在动作发生前已处于何种状态（已登录、购物车已有商品、第三方返回成功等）。 |
| **When** | **用户操作或系统触发动作**：具体一步交互或事件（点击按钮、提交表单、定时任务触发、收到回调等）。 |
| **Then** | **预期结果**：对外可观察的断言（HTTP 状态、页面文案、持久化状态、消息内容等），避免只写「正常」而无判据。 |

在 **正常、异常、边界、参数非法** 四类上各至少覆盖若干条；再按**当前业务领域**补充场景（支付、权限、库存等）。  
写入主表的格式与 **满减下单** 等完整 G/W/T 示例见 **[references/atdd-checklist-template.md](references/atdd-checklist-template.md)**（文末「完整示例」）。

**通用 + 领域验收细则（Then 判据查漏）**：以本包 **[references/acceptance-dimensions-general-index.md](references/acceptance-dimensions-general-index.md)** 为入口（同目录 `acceptance-*.md`）；**领域分类是否适用**见 [references/domain-system-focus-areas.md](references/domain-system-focus-areas.md)，**横切维度**见 [references/cross-cutting-coverage-dimensions.md](references/cross-cutting-coverage-dimensions.md)。以上均在本技能 `references/` 内自成体系，与验收场景 **做交叉引用**（一处写全、他处给链接或场景编号）即可，避免重复或矛盾。

### 3.1 验收文档交用户确认（闸门）

在 **第 3 节 ATDD 验收场景表**（及 **2.1 待确认问题** 中**阻塞项**已有答复或已标注「接受风险暂不澄清」）成稿后：

1. **2.1 逐项确认**可与文档起草**穿插进行**：未决的 `Q-ID` 按上节规则逐项收口径；**不得**在未答复的阻塞项上把 **Then** 写死成单一实现（除非已标注假设与风险）。  
2. **将整份验收文档交付用户 / 需求方**（路径、Issue、飞书文档等形式由项目约定），明确请对方审阅 **Given/When/Then** 与 **Then 判据** 是否与其期望一致。  
3. **取得明确结论后再进入第 4 节结构设计**：对方确认无修订，或已根据反馈**改表并再次确认**；将结论记入 **[references/atdd-checklist-template.md](references/atdd-checklist-template.md)** 中的 **「验收文档确认」** 表（时间、结论、是否可开发）。  
4. **禁止**在未经该确认（或用户明示跳过闸门）前，开始 **第 4 节 结构设计** 与后续产码；避免按错误 Then 白写实现。

### 4. 结构设计

**仅在 3.1 验收文档交用户确认（闸门）通过之后**，由验收点反推 **模块边界与接口**；先写清「谁负责什么」再写大段代码；**非必要不扩展**（YAGNI）。

### 5. ATDD 拆解 → TDD 内循环执行

1. **拆解验收场景为 TDD 任务**：为了让 ATDD 验收测试通过，识别需要哪些组件/类/方法
2. **逐任务执行 TDD 🔴红→🟢绿→重构循环**：
   - 🔴 **红**：为当前最小任务写一个单元测试，确认失败（非语法错误）
   - 🟢 **绿**：写极简代码使该单元测试通过
   - **重构**：优化代码结构，测试保持通过
3. **重复内循环**：直到所有为实现当前 ATDD 验收测试所需的小任务都完成
4. **全量 TDD 单元测试验证**：运行所有单元测试 → 全部通过

### 6. ATDD 验收场景全绿验证

**ATDD 外循环红绿节奏**：
- 🔴 **红**：初始 ATDD 验收测试失败（因为还没有实现任何代码）
- 🟢 **绿**：所有 TDD 内循环任务完成后，运行 ATDD 验收场景 → 全部通过

运行项目约定的全量测试命令（所有 TDD 单元测试/集成测试 + ATDD 验收场景）；
失败则回到步骤 5，识别遗漏的 TDD 任务，补完后再次验证。

声称完成前须**亲自执行**验证命令并得到通过输出（证据先于结论）。

## 三、自审（收工前）

- [ ] 任务清单是否已规划并逐项验证完成？
- [ ] 是否清晰理解 ATDD（外循环🔴红→🟢绿）与 TDD（内循环🔴红→🟢绿）的关系？
- [ ] ATDD 验收场景是否已正确拆解为多个 TDD 任务？
- [ ] 每条验收场景是否均含 **场景名称**、**场景 ID** 与 **Given / When / Then**，且前置、动作、预期**可执行、可断言**？
- [ ] **验收文档是否已交用户确认**并有记录；未通过确认前是否**未**进入结构设计？
- [ ] 若本轮改过主验收正文，是否在对应的 **`.execution-log.md`** 配套文件中**如实追加**了本会话实际对照过的路径？
- [ ] **待确认问题**是否已单独列表；每条是否含 **用户视角、可复制** 的澄清问句；已答复的是否已回填 **Then**？
- [ ] 对话中是否遵守 **逐项确认**，且在用户拍板一项后**自动带出下一项**（或明示队列已空）？
- [ ] 是否覆盖 **正常 / 异常 / 边界 / 参数非法** 与 **领域**？
- [ ] 是否先有结构设计再堆产码？
- [ ] 每个 TDD 任务是否**见过红再见绿**？
- [ ] 单测/集成是否按 **testing-style**（WHAT）与 **testing-anti-patterns**（Mock 禁区）写，而非锁实现细节或测 mock？
- [ ] ATDD 验收场景是否在当前分支**全绿**？
- [ ] 未决需求是否已 **提问** 或标注「待提供」？

## 四、刻意不做

- 用本技能名**替换** `npm test`、`test:all` 等仓库脚本名。  
- 无验收表即大段产码；或「表上写了 TDD」却从未**验红 / 验绿**。  
- **跳过 ATDD 直接做 TDD**：没有外循环导航，内循环可能走向错误终点。  
- **跳过 TDD 直接写实现**：没有可维护的底层实现，ATDD 会崩塌。
