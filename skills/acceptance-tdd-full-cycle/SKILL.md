# 验收测试驱动 + TDD 全周期
**先验收、后实现：验收文档成稿后先交用户确认，再动结构与产码；先单测/集成红绿，再外环自动化；未全绿不宣称完成。**

## 何时应用
- 从 spec / 需求说明 / 技术方案出发，要把功能**做到可验收、可回归**。
- 需要把「业务上做什么」与「代码上怎么证」绑在同一条链路。

## 工作流（按顺序，可选迭代）

### 1. 理解与澄清
通读 spec、需求说明、技术方案；对依赖、数据、环境等歧义点，**向业务方提问并记录结论**。

### 执行记录管理（硬性）
- 新建/实质修订主验收文档，不得在主文档内维护大表，必须在同名侧车文件 `*.execution-log.md` 追加记录。
- 侧车记录真实用到的文档路径，禁止重写未读文件、禁止让用户代填内容。

### 1.1 待确认问题（歧义 / 缺失 / 易争议）
- 需求歧义点单独成表，一行一问题，Q-ID 全程可追溯。
- 必须**用户视角提问**，不能只写技术实现视角。
- 每次只推进一项确认，用户答复后自动带出下一项待确认问题。
- 结论回填到验收场景 Then 与映射表，保证可落地自动化。

### 2. ATDD：验收测试点（Given / When / Then）
每条验收场景必备三项：
- **Given** 前置条件
- **When** 触发动作
- **Then** 可观察预期结果

同时要有唯一场景ID、业务可读场景名称。

### 2.1 验收文档用户确认（闸门）
阻塞性待确认问题闭环后，完整验收文档交付业务方审阅确认，**未经确认禁止进入结构设计与开发**。

### 3. 结构设计
闸门通过后，从验收点反推模块职责、接口契约与数据模型。

### 4. 产码 TDD
- 严格遵循 `references/tdd-core.md` 的「红-绿-重构」铁律。
- 断言必须遵循 `testing-style-observable-behavior.md` 的「可观察」原则，避免实现细节断言。

### 5. 外环自动化
- 按 `cross-cutting-coverage-dimensions.md` 补全横切维度（异常、边界、体验、运维）。
- 按 `domain-system-focus-areas.md` 完成领域重域（交易、订单、库存等）的专属风险覆盖。

## 核心产出
- `release-x-acceptance.md`：验收场景主表（Given/When/Then）
- `release-x-acceptance.execution-log.md`：执行记录侧车文件
- 单元/集成/E2E 用例：与验收场景一一映射

## 核心原则
1. **验收先行**：无确认的验收文档，不写一行代码。
2. **用户视角**：验收口径必须从用户/业务视角出发，而非纯技术视角。
3. **可观察断言**：Then 必须可被 HTTP/UI/DB/日志直接验证，禁止空话。
4. **全程追溯**：从需求澄清到自动化用例，全程可追溯、可审计。
