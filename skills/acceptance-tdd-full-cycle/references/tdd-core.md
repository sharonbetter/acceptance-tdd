# 产码侧 TDD 核心（本技能包内真源）

与「验收测什么」解耦：本节只约束 **写/改生产代码** 时的纪律。

---

## 铁律

**没有先失败的测试，就不写为通过该测试而存在的生产代码。**

已实现、尚无测试 → 先补**能描述期望行为**的用例；若一跑就绿，说明漏测或未测到新行为，**改测例**直到在错误实现上**失败**，再改实现。

---

## 红 → 验红 → 绿 → 验绿 → 重构

1. **红**：写**一条最小用例**，表达尚未满足的行为；运行测试，**必须失败**，且失败原因符合预期（缺能力、断言失败），非语法噪声。
2. **验红**：未见到失败，**不写**实现。
3. **绿**：只写**最少**代码使该用例通过。
4. **验绿**：相关测试与约定全量命令仍绿。
5. **重构**：在仍绿前提下整理结构；**不**在重构步增加新行为（新行为 = 新红）。

---

## 好测与坏测（摘要）

- **好测**：名称说清行为；断言可观察结果；单用例单意图；优先真实协作，mock 仅用于系统边界。
- **坏测**：测 mock 调用次数；名称含糊；与实现细节强耦合导致一改就碎。

---

## 常见借口（一律驳回）

| 借口 | 回应 |
| --- | --- |
| 太简单不用测 | 简单逻辑也会回归；用例成本通常低于事后排障。 |
| 写完再补测 | 后补测一跑就绿，无法证明测到了「应有行为」。 |
| 先留着实现当参考 | 等于测后补写；删除或搁置，从红开始。 |
| 手动点过就行 | 不可重复、不可回归；压力一大必漏。 |

---

## 收工前检查

- [ ] 每条新行为是否都经历过 **可见的失败** 再变绿？
- [ ] 是否用项目约定命令跑过 **相关测 + 外环测**？
- [ ] 控制台/日志是否无意外错误（已知弃用类警告单独说明）？
