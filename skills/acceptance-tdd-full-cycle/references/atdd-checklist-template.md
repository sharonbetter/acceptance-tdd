## 逐项确认（对话节奏，代理须遵守）

与用户在聊天中推进验收时：

1. **一次只推一项**：用**一小段用户故事 + 明确选项或判据**呈现当前 `Q-ID`；不要一次抛整张表让用户逐格回复。  
2. **确认后依次三步**：用户给出结论后，代理须 **更新本文档**（本 `Q-ID` 行、映射表、受影响场景的 **Then**；阻塞项解决后勾掉或移入「已决议」）→ **简短复述已定结论** → **立即**给出**下一项** `Q-ID` 的同一格式问句。  
3. **队列顺序**：默认 **表 A** 按 `A-01, A-02, …` → **表 B** 按 `B-01, B-02, …`；若某 `B-xx` 依赖 `A-xx`，在表 B 行备注「依赖 A-xx」，对话中在 `A-xx` 定案后再问该项。  
4. **队列为空**：说明「待确认项已清」，并请用户进行 **「验收文档确认」闸门**（见上表）的整表确认（或按项目约定的一键确认句）。  
5. **用户答复格式**：鼓励 `Q-ID：结论`（如 `B-01：改为方案 2`），便于审计与脚本化回填。

## 附录：完整示例（满减下单）

以下为 **填好的示例**，演示 GWT 粒度；实际项目请替换业务名词与数值。

### 场景：满减后实付正确

| 场景 ID | 场景名称 | 分类 | Given | When | Then | 自动化 |
|---------|----------|------|--------|------|------|--------|
| S-PROMO-01 | 满减后实付金额正确 | 成功 | 用户已登录；购物车含 A×2（单价 100）、B×1（单价 50）；活动「满 200 减 30」生效 | 用户提交订单并支付 | 订单实付 = 200 − 30 = **170**；订单明细行金额与活动分摊可核对；库存相应扣减 | E2E + 计价单元 |

**Given（展开）**  
- 用户 `u1` 已登录，会话有效。  
- 商品 A 库存 ≥2，B 库存 ≥1。  
- 活动规则：购物车**小计**（不含运费）≥200 时减 30；与券可叠加以 spec 为准（本例假设不叠券）。

**When**  
- 将 A×2、B×1 加入购物车 → 进入结算 → 确认订单 → 完成支付（或 mock 支付成功回调，以 spec 为准）。

**Then**  
- 支付请求或订单详情中 **应付总额 = 170**（若 spec 规定运费另计，则写清「商品小计 170 + 运费 X = 总价 Y」）。  
- 返回或页面上 **无 5xx / 未处理异常**；错误路径另有场景表项。  
- 库存：A 减 2，B 减 1（若验收范围包含库存）。

**Then（反例，勿写）**  
- ~~系统正确处理满减逻辑~~（不可执行、不可测）
