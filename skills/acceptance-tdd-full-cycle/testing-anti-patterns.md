# 测试反模式（禁止使用）

**目的**：识别并避免会导致测试脆弱、不可靠、维护成本高的写法。

---

## 1. 过度 Mock
- 表现：把所有依赖都 mock，测试里几乎没有真实代码。
- 危害：用例只验证 mock 调用，不验证真实业务逻辑，改实现就碎。
- 修正：mock 仅用于系统边界（外部服务、数据库），优先真实协作。

## 2. 断言实现细节
- 表现：断言内部方法调用、私有变量、mock 调用次数。
- 危害：实现一改，用例就全红，测试变成维护负担。
- 修正：按 `testing-style-observable-behavior.md` 改为断言外部可观察结果。

## 3. 大而全的用例
- 表现：一个用例测多个行为、多个场景，步骤冗长。
- 危害：用例失败无法定位问题，维护成本高。
- 修正：拆成单意图用例，一个用例只测一个行为/场景。

## 4. 依赖执行顺序
- 表现：用例 A 的结果依赖用例 B 的执行。
- 危害：单独跑或并行跑时，用例会随机失败。
- 修正：每个用例都做完整的前置准备，保持独立。

## 5. 重复断言
- 表现：在多个用例里重复断言同一个基础逻辑（如登录）。
- 危害：维护成本高，修改时需要改多个地方。
- 修正：把基础逻辑抽到 setup 或公共 fixture 里。
