# 测试风格指南：断言可观察行为

**目的**：让测试只依赖系统的**外部可观察结果**，而非内部实现细节，从而做到「改实现不改用例」。

---

## 铁律
1. **只断言行为，不断言实现**：
   - 好：断言 HTTP 状态码、响应体字段、UI 文案、数据库状态、日志指标。
   - 坏：断言内部方法调用、mock 调用次数、私有变量值、实现分支细节。
2. **断言必须唯一可验证**：Then 必须明确，不能有「多种实现都能通过」的歧义。
3. **避免过度指定**：不写超出业务契约的细节断言，比如字段顺序、无关字段值。

---

## 断言示例
| 场景 | 好断言（可观察） | 坏断言（实现细节） |
| --- | --- | --- |
| 登录 | 断言 200、token 存在、用户信息正确 | 断言 `UserService.login()` 被调用 |
| 下单 | 断言订单状态为「已支付」、库存扣减正确 | 断言 `PaymentProcessor.process()` 被调用 |
| 权限 | 断言 403 状态码、「无权限」文案 | 断言 `PermissionChecker.check()` 返回 false |

---

## 命名规范
- 用例名格式：`[行为]_[场景]_[预期结果]`
- 例：`当用户A访问用户B的订单时_返回403`
- 禁止：`testOrderAccess1`、`testPermission` 这类模糊命名。
