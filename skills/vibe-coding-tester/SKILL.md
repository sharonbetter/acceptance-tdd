# Vibe-Coding Tester

## 背景与目标

Vibe-coding 场景下，开发者通过与 AI 的多轮对话来迭代代码，往往没有正式的 spec 文档。但对话本身就是最真实的"需求记录"——里面藏着业务目标、边界约束、修正历史和踩过的坑。

这个技能的核心思路是：**先读懂对话，再反向推导 Spec，最后生成测试用例**。

整个流程分两个阶段：
1. **阶段一**：分析对话 → 输出《反向推导 Spec 清单》，等用户确认
2. **阶段二**：基于确认后的 Spec → 生成测试用例

---

## 阶段一：反向推导 Spec

### 输入来源判断

首先判断对话内容从哪里来，按以下优先级处理：

**① 用户已提供内容**（直接分析，无需额外操作）
- 用户在当前对话中粘贴了对话文本 → 直接分析粘贴内容
- 用户上传了对话记录文件（.txt / .md） → 读取文件后分析
- 当前对话本身就是 vibe-coding 对话 → 分析当前 session 历史消息

**② 用户未提供内容，但说明了使用的工具**（主动帮用户提取）

根据工具类型，按以下方式获取对话记录：

**Cursor**
Cursor 的对话历史存储在本地 SQLite 数据库中。运行以下命令提取当前工作区最近一个 Chat tab 的对话：
```bash
# 找到当前工作区对应的 vscdb（取最近修改的）
DB=$(ls -t ~/Library/Application\ Support/Cursor/User/workspaceStorage/*/state.vscdb 2>/dev/null | head -1)

# 提取所有 Chat tab 的对话文本
sqlite3 "$DB" "SELECT value FROM ItemTable WHERE key='workbench.panel.aichat.view.aichat.chatdata';" \
  | python3 -c "
import sys, json
d = json.load(sys.stdin)
for tab in d.get('tabs', []):
    print(f\"=== Tab: {tab.get('chatTitle','无标题')} ===\")
    for b in tab.get('bubbles', []):
        role = '用户' if b.get('type') == 'user' else 'AI'

        text = b.get('text', '')
        if text.strip():
            print(f'[{role}]: {text[:500]}')
    print()
"
```
如果有多个 tab，询问用户："找到 N 个对话 tab，分别是：[列出标题]，请问要分析哪一个？"

Cursor 的 **Composer**（Agent 模式）对话存储在同一数据库的 `composer.composerData` key 下，结构类似，bubbles 字段同样包含 `type`（user/ai）和 `text`。

**Claude.ai / Claude Desktop**
Claude 的对话不存储在本地文件系统，需要用户手动提供。提示用户：
> "请在 Claude.ai 或 Claude Desktop 中，打开你的对话，全选复制（Cmd+A → Cmd+C），然后粘贴到这里。"

**CatDesk（当前工具）**
如果用户说"就是我们现在这个对话"，直接分析当前 session 的历史消息即可，无需额外操作。

**其他 Code Agent（Windsurf、Copilot Chat、通义灵码等）**
这类工具通常不提供本地导出接口。提示用户：
> "请在 [工具名] 中找到对话窗口，手动复制对话内容粘贴过来。通常可以全选对话区域后复制，或者工具菜单里有'导出对话'选项。"

**③ 用户什么都没提供，也没说用什么工具**

询问用户：
> "请告诉我你用的是哪个工具开发的（Cursor / Claude / 其他），我来帮你提取对话记录；或者直接把对话内容粘贴过来也可以。"

### 分析维度

阅读对话时，**完全忽略具体的代码实现细节**（函数名、变量名、具体语法），只关注以下四个维度：

#### 1. 核心业务目标（Happy Path）
- 这个功能/工具到底是干什么的？
- 用户最初的诉求是什么？
- 最终实现的主流程是什么？

#### 2. 隐藏的业务边界与约束
从对话中寻找用户说过的"不要这样"、"调整一下"、"注意 XX 情况"、"应该是 XX 而不是 XX"等修正性表达，提取出：
- 输入的合法范围（什么是有效输入，什么是无效输入）
- 输出的格式要求（格式、精度、顺序等）
- 特殊情况的处理规则（边界值、空值、异常状态）
- 用户明确拒绝的行为（"不要 XX"、"别 XX"）

#### 3. 痛苦点与历史 Bug（Pain Points）
从对话中寻找报错信息、"又出问题了"、"还是不对"、"死循环"、"逻辑混乱"等表达，提取出：
- 曾经出现过的 bug 或错误
- 修复过的逻辑问题
- 反复调整的地方（说明这里容易出错）

#### 4. 隐含的非功能性要求
- 性能期望（"太慢了"、"要快一点"）
- 用户体验要求（"提示不够清晰"、"交互太麻烦"）
- 兼容性要求（"在 XX 环境下也要能用"）

### Spec 清单输出格式

输出一份结构化的《反向推导 Spec 清单》，格式如下：

---

## 📋 反向推导 Spec 清单


### 功能概述
[一句话描述这个功能是干什么的]

### 核心业务目标（Happy Path）
[描述主流程：用户做了什么 → 系统做了什么 → 最终结果是什么]

### 业务边界与约束规则
| 约束类型 | 规则描述 | 来源（对话中的原话或意图） |
|---------|---------|--------------------------|
| 输入约束 | ... | "用户说：..." |
| 输出格式 | ... | "用户说：..." |
| 特殊情况 | ... | "用户说：..." |
| 禁止行为 | ... | "用户说：..." |

### 历史 Bug 与风险点
| 问题描述 | 触发条件 | 是否已修复 |
|---------|---------|-----------|
| ... | ... | 是/否/未知 |

### 非功能性要求
[如有，列出性能、体验、兼容性等要求；如无，写"暂无明确要求"]

### ⚠️ 信息缺口（需用户确认）
[列出对话中没有明确说明、但测试时需要知道的信息，例如：
- "当输入为空时，期望行为未明确"
- "最大支持的数据量未提及"]

---

输出完 Spec 清单后，**停下来等待用户确认**，说：

> "以上是我从对话中推导出的 Spec 清单。请确认：
> 1. 有没有理解错误的地方？
> 2. 有没有遗漏的重要规则？
> 3. ⚠️ 信息缺口 部分，你能补充一下吗？
>
> 确认后我会基于这份 Spec 生成测试用例。"

---

## 阶段二：生成测试用例

用户确认 Spec 后（可能有修改），基于最终版 Spec 生成测试用例。

### 测试用例覆盖策略

按以下优先级覆盖：

1. **Happy Path 用例**：验证核心业务目标能正常完成
2. **边界值用例**：针对每条"业务边界与约束规则"设计正向和反向用例
3. **历史 Bug 回归用例**：针对每个"历史 Bug 与风险点"设计专项回归用例（优先级高，因为这些是已知的坑）
4. **异常/错误处理用例**：无效输入、空值、极端情况
5. **非功能性用例**（如有明确要求）

### 测试用例输出格式

默认使用 Markdown 表格格式输出，除非用户指定其他格式（CSV、Excel、JSON 等）。
```markdown
## 测试用例清单

### 模块：[功能名称]

| 用例ID | 用例名称 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 类型 |
|--------|---------|---------|---------|---------|--------|------|
| TC-001 | [Happy Path] 正常完成主流程 | ... | 1. ... 2. ... | ... | P0 | 功能 |
| TC-002 | [边界] 输入为空时的处理 | ... | ... | ... | P1 | 边界 |
| TC-003 | [回归] 修复过的 Bug X 不再复现 | ... | ... | ... | P0 | 回归 |
```

**类型标签说明**：
- `功能`：Happy Path 和正向功能验证
- `边界`：边界值和约束规则验证
- `回归`：针对历史 Bug 的专项回归
- `异常`：错误输入和异常状态处理
- `非功能`：性能、体验等

### 输出后的询问

生成测试用例后，询问用户：
> "测试用例已生成，共 X 条。需要我帮你导出成其他格式（CSV、Excel、Markdown 文件），或者直接在这里使用就好？"

---

## 注意事项

- **始终忽略代码实现细节**：测试用例描述的是业务行为，不是代码逻辑。用"当用户输入 X 时"而不是"当函数 foo() 被调用时"。
- **保留对话原话作为证据**：在 Spec 清单的"来源"列中，尽量引用对话中的原话，让用户能对应上。
- **历史 Bug 用例优先级最高**：已经踩过的坑最容易再次踩，回归用例要标 P0。
- **信息缺口要明确列出**：不要猜测，不要假设，把不确定的地方暴露出来让用户补充。
